1. CURRENT STATE (what already works)
================================================================================

The Madness effect forces Mobs (server-side AI) and Players (client-side bot) into blind, uncontrolled rage. Already implemented:

  A) SERVER-SIDE MOB TARGETING:
     - Non-boss mobs under Madness attack any nearby living entity regardless of faction.
     - Targets cycle erratically (simulating hysteria).
     - Boss filtering exists: ModCompatManager.isBoss(mob) restricts boss madness targets
       to Players, other Bosses, and Elite mobs (medallion carriers).
     - LivingChangeTargetEvent handler caches valid targets in NBT (xebMadnessTargetId)
       to prevent infinite target-reset loops.

  B) CLIENT-SIDE PLAYER BOT (MadnessClientHandler.java):
     - Intercepts player input under Madness effect.
     - Dynamic weapon selection based on target distance:
       * Ranged range (> 5.0 blocks): selects bows/crossbows.
       * Melee range (<= 5.0 blocks): selects swords/axes.
     - Weapon scoring algorithm:
       * Base Score = Attack Damage attribute + Tier damage bonus.
       * Keyword bonus: +15.0 for registry names containing sword, blade, claymore, katana, incinerator, etc.
       * Pure ranged penalty: -20.0 in melee range for bows/crossbows.
       * Hybrid exemption: weapons with right-click ranged charge (like The Incinerator) are exempt from ranged penalty in melee.
     - Melee swing: triggers keyAttack and triggerClientAttack when cooldown >= 0.9.
     - Better Combat partial support: sets keyAttack.setDown(true) for combo triggers.
     - Anti-stick fixes:
       * Target loss → releases keyUse/keyAttack/activeItem.
       * Slot swap → resets hybridChargeTicks, releases keyUse/activeItem.
       * Out-of-melee charge canceling for hybrid weapons.
       * Melee right-click interrupt (forces keyUse false unless hybrid special charging).

  C) CONFIG: A working ForgeConfigSpec config file already exists with parameters
     for ranges, scores, penalties, etc.

2. PROBLEMS TO SOLVE
================================================================================

PROBLEM 1 — MODDED WEAPONS NOT DETECTED PROPERLY (Players)
The current weapon scoring relies almost exclusively on registry name keyword matching (sword, blade, katana, etc.) and vanilla attributes. This fails for:
  - Mods that use custom registry names (e.g., "epicfight:iron_greatsword" works, but
    "some_mod:frost_reaper" or "tetra:modular_sword" may not match any keyword).
  - Mods that add weapons via capabilities/tags instead of standard attributes.
  - Weapons from Iron Spells, Artifacts, Appli ed Energistics 2 tools, Tinkers'
    Construct, Tetra, and similar mods that use non-standard item classes.
  - Result: The player bot holds a modded weapon but doesn't recognize it as usable,
    stands still, or ignores it in favor of a weaker vanilla item.

PROBLEM 2 — BOSSES (CATACLYSM & OTHER MODS) DON'T ATTACK UNDER MADNESS
Current behavior:
  - Cataclysm bosses (Ignis, Netherite Monstrosity, Ancient Remnant, etc.) and similar
    bosses from other mods either freeze in place or ignore Madness entirely.
  - The isBoss() check exists but the target validation pipeline is too restrictive.
  - Many modded bosses have custom AI goal systems (not standard targetSelector) that
    override or ignore the LivingChangeTargetEvent handler.
  - Some bosses use custom phase systems (Cataclysm's Ignis has phase transitions) where
    the boss only attacks players and ignores all other targets regardless of events.
  - Result: Bosses stand idle until a player walks close enough, completely defeating
    the purpose of the Madness effect.

PROBLEM 3 — BETTER COMBAT INTEGRATION IS INCOMPLETE
Current behavior:
  - The bot sets keyAttack.setDown(true) but doesn't account for Better Combat's
    specific attack patterns, combo windows, or weapon-specific timing.
  - Better Combat assigns different attack speeds, combo lengths, and special abilities
    per weapon. The current bot ignores these and uses a generic cooldown threshold (0.9).
  - The bot doesn't detect when Better Combat is actually modifying a weapon's behavior
    vs. when vanilla combat is active for the same weapon.
  - Result: Attacks feel inconsistent — sometimes the bot attacks too fast, sometimes
    too slow, and special weapon abilities from Better Combat are never triggered.

3. REQUIREMENTS — UNIVERSAL WEAPON DETECTION SYSTEM
================================================================================

Build a modular, extensible weapon classification system that correctly identifies
weapons from ANY mod, not just vanilla and keyword-matched items.

3.1 WEAPON CLASSIFICATION RESULT
Each scanned ItemStack must resolve to ONE of these categories:
  - MELEE: Swords, axes, daggers, spears, gauntlets, and any close-range weapon.
  - RANGED: Bows, crossbows, staves (that shoot projectiles), and any far-range weapon.
  - HYBRID: Weapons that can do both (e.g., The Incinerator — melee sword with
    right-click ranged charge).
  - MAGIC: Weapons that cast spells or shoot magic projectiles (may overlap with RANGED).
  - TOOL: Pickaxes, shovels, etc. — usable as emergency melee but lowest priority.
  - NON_WEAPON: Items that should never be selected (food, blocks, potions, etc.).

3.2 DETECTION PIPELINE (ordered by priority, first match wins)

  LAYER 1 — TAG-BASED DETECTION (highest priority):
  Check for Forge tags (or vanilla tags) on the item:
    - #minecraft:swords, #minecraft:axes, #minecraft:pickaxes (vanilla tags)
    - Any tag from the Better Combat mod if loaded: #bettercombat:swords, etc.
    - Any mod-specific weapon tags. Build a registry that maps known mod namespaces
      to their weapon tag conventions. Examples:
        * "epicfight:" → check Epic Fight's sword/axe/greatsword tags
        * "irons_spells:" → check for spell weapon tags
        * "tetra:" → check for modular weapon tags
        * "tconstruct:" → check for Tinkers' melee/ranged tool tags
        * "artifacts:" → check for artifact weapon tags
        * "apotheosis:" → check for affix weapon tags
  If the item has a recognized weapon tag → return the corresponding WEAPON_CLASS.

  LAYER 2 — ATTRIBUTE-BASED DETECTION:
  If no tag match, inspect the item's attributes:
    - If item has ATTACK_DAMAGE modifier with value > 0 → candidate for MELEE or HYBRID.
    - If item has ATTACK_SPEED modifier:
        * speed >= 1.6 → likely a fast melee weapon (sword/dagger).
        * speed < 1.6 but > 0 → likely a slow heavy weapon (axe/greatsword).
    - If item implements ShootableItem or has RANGED_ATTACK_DAMAGE → RANGED or HYBRID.
    - If item has both ATTACK_DAMAGE and is ShootableItem → HYBRID.
  Calculate a confidence score. If confidence >= threshold (configurable) → return class.

  LAYER 3 — ITEM CLASS / INTERFACE DETECTION:
  If no tag/attribute match, check the item's Java class hierarchy:
    - SwordItem, AxeItem, DiggerItem → MELEE.
    - BowItem, CrossbowItem, ShootableItem → RANGED.
    - Check for mod-specific base classes:
        * "net.titanium.item.SwordItem" (Titanium)
        * "dev.xkmc.l2weaponry.content.item.MeleeWeapon" (L_Ender's Cataclysm weapons)
        * "yesman.epicfight.api.data.reloader.item.WeaponCategory" checks (Epic Fight)
        * Other known mod weapon base classes.
  If matched → return the corresponding WEAPON_CLASS.

  LAYER 4 — REGISTRY NAME HEURISTICS (lowest priority, current system as fallback):
  Use the existing keyword matching system as the final fallback layer.
  Expand the keyword list significantly with entries from popular mods:
    - Existing: sword, blade, claymore, katana, incinerator
    - Add: rapier, spear, halberd, greataxe, hammer, maul, dagger, scythe, glaive,
      pike, cutlass, saber, scimitar, cleaver, fist, gauntlet, claw, tonfa,
      staff, wand, tome, grimoire, spellbook, bow, crossbow, longbow, shortbow,
      repeater, railgun, musket, pistol, blaster, cannon, launcher, throwstar,
      shuriken, boomerang, chakram, disc, trident
  Apply a LOWER confidence score for this layer so attribute/tag matches always win.

  LAYER 5 — CAPABILITY CHECK (for mods that use ForgeCapabilities):
  Some mods store weapon data in capabilities attached to ItemStacks rather than
  attributes or tags. Check for known capability providers:
    - Epic Fight's weapon capability
    - Better Combat's weapon style capability
    - Tetra's modular item capability
  If a capability reports the item as a weapon → use that classification.

  LAYER 6 — FALLBACK:
  If NO layer classifies the item → NON_WEAPON.

3.3 WEAPON SCORING v2 (replaces current scoring)
  For each hotbar slot with a classified weapon, calculate:

    score = baseDamage                                    // from ATTACK_DAMAGE attribute
           + tierBonus                                    // wood=0, stone=1, iron=2, diamond=3, netherite=4, mod_tier=mapped
           + keywordBonus                                 // +15.0 for name heuristics (LAYER 4)
           + classificationBonus                         // MELEE gets +5 in melee range, RANGED gets +5 in ranged range
           + specialAbilityBonus                         // +10.0 if weapon has known special abilities (fire aspect, knockback, etc.)
           + enchantmentBonus                            // sum of relevant enchantment levels * 2.0
           - rangedPenalty                               // -20.0 for pure RANGED in melee range (existing)
           - cooldownPenalty                             // -currentCooldown * 3.0 (don't pick weapons on long cooldown)
           + betterCombatComboBonus                      // +8.0 if Better Combat is loaded and weapon has combo>1

  Tier mapping for mod materials should be configurable. Create a tier mapping:
    - Map common mod material names to tier values (e.g., "adamantine"→5, "dragonforged"→5,
      "cobalt"→3, "manyullyn"→4, "queens_slime"→4, etc.)

3.4 DYNAMIC RANGE THRESHOLD
  The current hardcoded 5.0 block threshold for melee/ranged switch is too rigid.
  Make it weapon-dependent:
    - For each classified weapon, calculate an effective reach:
        reach = player.getAttributeValue(Attributes.BLOCK_INTERACTION_RANGE)
              + (weapon is MELEE ? 0.5 : 0)
              + (Better Combat loaded ? getExtendedReach(weapon) : 0)
    - Use this reach value instead of hardcoded 5.0 for melee/ranged decision.
    - Add a configurable buffer zone (e.g., +1.5 blocks) where the bot evaluates
      both options and picks the higher-scored weapon.

4. REQUIREMENTS — BOSS AI OVERHAUL (Cataclysm & Others)
================================================================================

The current boss targeting system is too passive. Many modded bosses have custom AI
that ignores standard Minecraft targeting events. We need a multi-layered approach.

4.1 UNIVERSAL BOSS DETECTION (improve isBoss())
  Expand boss detection beyond the current implementation:

  CHECK 1 — Existing: ModCompatManager.isBoss(mob) check (keep this).
  CHECK 2 — Health threshold: mob.getMaxHealth() >= configurable threshold (default: 300).
  CHECK 3 — Boss bar detection: Check if the mob has a BossEvent registered
            (Forge's BossEventManager or vanilla's BossInfo).
  CHECK 4 — Category check: mob.getType().getCategory() == EntityType.CATEGORY_MISC
            with high health (many boss mods use MISC category).
  CHECK 5 — Mod namespace blacklist: Entities from known boss mod namespaces
            (cataclysm, alexsmobs, mutagen, etc.) are flagged as potential bosses.
  CHECK 6 — Capability/interface: Check if the entity implements known boss interfaces
            from mods (e.g., Cataclysm's boss marker interface).

  An entity is considered a "boss" if CHECK 1 OR (CHECK 2 AND (CHECK 3 OR CHECK 4 OR CHECK 5)).

  IMPORTANT: Make the boss detection result cacheable per entity type (not per entity
  instance) to avoid re-checking every tick. Cache for 100 ticks, invalidate on
  mod load.

4.2 AGGRESSIVE TARGET INJECTION (NEW — for frozen bosses)
  For bosses that IGNORE the standard LivingChangeTargetEvent (like Cataclysm bosses
  in certain phases), implement a DIRECT goal injection system:

  STEP 1 — Detect frozen state:
    If a boss under Madness has had the same target for > 60 ticks AND
    has not dealt damage in that time AND
    is not currently performing a phase transition animation (detect via
    movement = 0 AND attack animation not playing)
    → mark as "frozen" in a debug map.

  STEP 2 — Force-move to nearest valid target:
    When a boss is marked "frozen", bypass ALL AI:
    - Clear the boss's goalSelector temporarily.
    - Add a custom PriorityGoal (highest priority) that:
        * Moves the boss directly toward the nearest valid Madness target
          (Player/Boss/Elite within detection range).
        * Uses moveControl.setWantedPosition() to force movement.
        * When within attack range, calls mob.lookAt(target) and
          mob.swing() to force an attack animation.
    - After 20-40 ticks of forced movement, restore the original goalSelector
      and let the boss AI take over again.

  STEP 3 — Attack execution fallback:
    If the boss's AI still refuses to attack after forced movement:
    - Use reflection or access transformers to call the boss's melee/ranged
      attack method directly (if detectable).
    - Fallback: Deal damage directly via target.hurt() with the boss's
      attack damage attribute. This should be a LAST RESORT and only if
      the config option "forceAttackOnFrozenBoss" is enabled.

  STEP 4 — Phase transition awareness:
    For mods like Cataclysm that have phase transitions (Ignis flies up,
    becomes immune, etc.), detect phase transitions by:
    - Checking if the boss is invulnerable (boss.isInvulnerableTo(source)).
    - Checking if the boss has a "phase" data in its NBT or capability data.
    - During phase transitions, DO NOT inject forced targets — let the boss
      complete its transition, then resume Madness targeting.

4.3 TARGET CANDIDATE EXPANSION FOR BOSSES
  Currently, boss Madness targets are restricted to Players, Bosses, and Elites.
  This is too restrictive for some scenarios. Make it configurable:
  - Default: Players + Bosses + Elites (current behavior).
  - Config option "bossAttackAllMobs" (default: false) — if true, bosses under
    Madness attack ANY living entity like non-boss mobs do.
  - When a boss is "frozen" and the restricted candidate set is empty (no players,
    bosses, or elites nearby), TEMPORARILY expand to all living entities for 100
    ticks to prevent permanent freezing.

4.4 ANTI-REPEATED-TARGET LOOP
  The existing NBT cache (xebMadnessTargetId) prevents infinite loops when the AI
  rejects targets. Improve it:
  - Track the LAST 5 rejected target IDs per entity (ring buffer in NBT).
  - When selecting a new target, exclude all targets in the rejection buffer.
  - Clear the rejection buffer every 200 ticks to prevent candidate exhaustion.

5. REQUIREMENTS — BETTER COMBAT FULL INTEGRATION
================================================================================

Better Combat (by daedelus_dev) is a combat overhaul mod that changes how weapons
work in Minecraft. The Madness bot must FULLY integrate with its systems.

5.1 DETECT BETTER COMBAT WEAPON STYLES
  Better Combat assigns a "WeaponStyle" to each weapon via JSON configs.
  Each style defines:
    - attackRange: The effective attack range (replaces vanilla 3.0).
    - comboLength: How many hits are in a combo sequence.
    - attackSpeed: Per-weapon attack speed (may differ from attribute).
    - specialAbilities: List of abilities (thrust, sweep, leap, etc.).
    - twoHanded: Whether the weapon requires two hands.

  Access Better Combat's weapon style data:
    - If BC is loaded, use the BC API to query the weapon style for each ItemStack.
    - If the BC API is not available, use reflection to access:
        "dev.xkmc.bettercombat.client.ClientData"
        or the server-side weapon style registry.
    - Cache weapon styles per item type (not per stack) for performance.

5.2 ADAPTIVE ATTACK TIMING
  Replace the generic "cooldown >= 0.9" check with weapon-aware timing:

  FOR EACH WEAPON (when BC is loaded):
    - Get the weapon's comboLength and attackSpeed from BC's WeaponStyle.
    - Track the current combo step (0 to comboLength-1).
    - Each combo step may have a different timing window.
    - Call keyAttack.press() at the OPTIMAL moment:
        * If the weapon has combo steps, time the press to the combo window.
        * If no combo (single-hit weapon), use the vanilla cooldown threshold.
    - After the last combo hit, wait for the full cooldown before restarting.

  FOR VANILLA COMBAT (when BC is NOT loaded):
    - Keep the existing cooldown >= 0.9 logic.
    - This is the fallback path that must always work.

5.3 SPECIAL ABILITY TRIGGERING
  Better Combat weapons can have special abilities triggered by specific inputs:
  - LEAP: Triggered by jumping + attacking. The bot should auto-jump before attacking
    when the weapon has a leap ability and the target is 3-6 blocks away.
  - SWEEP: Automatic on attack, no special handling needed.
  - THRUST: The bot should sprint toward the target before attacking for thrust weapons.
  - PARRY_STANCE: If the weapon has a parry ability (right-click), the bot should
    use keyUse briefly when the target is about to attack (heuristic: target is
    facing the bot and within attack range).
  - Any other special abilities defined by the weapon's style should be detected
    and triggered appropriately via the bot's input simulation.

5.4 TWO-HANDED WEAPON HANDLING
  When the bot selects a two-handed weapon (from BC's weapon style):
  - Automatically move the offhand item to a non-weapon hotbar slot if it contains
    a weapon (to avoid the two-handed penalty in some modpack configurations).
  - Do NOT drop or destroy items — just swap slots.

5.5 FALLBACK WHEN BC IS NOT LOADED
  The bot must work perfectly with OR without Better Combat installed.
  Use a ModList.get().isLoaded("bettercombat") check and maintain two code paths:
  - BC loaded → use BC-aware weapon styles, combo timing, special abilities.
  - BC not loaded → use vanilla attribute-based combat (existing system).

6. REQUIREMENTS — HOTBAR PRIORITY ENGINE v2
================================================================================

The hotbar scanning and weapon switching system needs to be more intelligent.

6.1 HOTBAR SCANNING FREQUENCY
  - Scan hotbar every 10 ticks (0.5 seconds), NOT every tick (performance).
  - Force an immediate rescan when:
    * The player picks up a new item.
    * The player's attack cooldown completes.
    * The current weapon breaks (durability reaches 0).
    * The target distance crosses the melee/ranged threshold.

6.2 INTELLIGENT SLOT SWITCHING
  When switching to a new weapon slot:
  - Pre-release any active item use (existing behavior).
  - Reset hybridChargeTicks (existing behavior).
  - NEW: If the previous weapon was being used in a Better Combat combo,
    wait for the combo to finish or the cooldown to expire before switching.
    Exception: If the new weapon's score is 20+ points higher (emergency upgrade),
    switch immediately even mid-combo.

6.3 AMMO/ARROW AWARENESS
  When the bot selects a ranged weapon:
  - Check if the player has arrows/ammo in inventory (not just hotbar).
  - If no ammo is available, penalize the ranged weapon score by -50.0.
  - If the offhand has a non-arrow item and the hotbar has arrows in another slot,
    auto-swap the arrow to the offhand.

6.4 DURABILITY AWARENESS
  - Do not select weapons with 0 durability.
  - Penalize weapons below 10% durability by -5.0 (risk of breaking mid-combat).
  - If the currently held weapon breaks mid-combat, immediately rescan and switch.

7. ARCHITECTURE & DESIGN PATTERNS
================================================================================

Use these patterns to keep the code maintainable and extensible:

7.1 STRATEGY PATTERN — Weapon Detectors
  Create an interface:
    WeaponClassifier {
        WeaponClass classify(ItemStack stack);
        double confidence(ItemStack stack);  // 0.0 to 1.0
    }
  Each detection layer (Tag, Attribute, Class, Heuristic, Capability) is a
  separate implementation. A WeaponClassificationEngine runs them in priority
  order and returns the highest-confidence result.

7.2 ADAPTER PATTERN — Mod Compatibility
  Create a ModCompatAdapter interface:
    ModCompatAdapter {
        String modId();
        boolean isLoaded();
        WeaponClass classifyWeapon(ItemStack stack);
        boolean isBoss(LivingEntity entity);
        Optional<WeaponStyle> getWeaponStyle(ItemStack stack);  // For BC integration
        boolean isItemNonWeapon(ItemStack stack);
    }
  Each mod gets its own adapter. The adapter is only instantiated if the mod is loaded.
  This makes adding new mod support as simple as adding one new adapter class.

7.3 OBSERVER PATTERN — Target Tracking
  Instead of polling for targets every tick, use event-driven target tracking:
  - Listen to EntityJoinLevelEvent, EntityLeaveLevelEvent for candidate management.
  - Listen to LivingDamageEvent to detect when targets die.
  - Invalidate cached target lists when entities enter/leave the tracking range.

7.4 STATE MACHINE — Player Bot States
  The player bot should operate as a state machine with clear states:
    IDLE → SCANNING → APPROACHING → ATTACKING_MELEE → ATTACKING_RANGED →
    RETREATING → SWITCHING_WEAPON → SPECIAL_ABILITY → FROZEN_RECOVERY
  Each state has defined entry/exit conditions and behavior. This prevents the
  current "spaghetti if/else" approach and makes debugging easier.

7.5 CACHE MANAGEMENT
  - Cache weapon classifications per Item type (not per ItemStack) with 100-tick TTL.
  - Cache boss detection results per EntityType with 600-tick TTL.
  - Cache Better Combat weapon styles per Item with 200-tick TTL.
  - Use WeakHashMap or explicit cache clearing on world unload.

  - NEVER access mod internals via reflection without a fallback path.
  - If a mod's adapter fails to load or throws an exception, log it and fall
    back to the next detection layer. The system must NEVER break vanilla gameplay.

12. CONSTRAINTS & OPTIMIZATION RULES
================================================================================

PERFORMANCE:
  - ZERO allocations per tick in hot paths (weapon scanning, target selection).
  - Cache everything that can be cached. Use object pools for frequently created objects.
  - The classification engine should classify all 9 hotbar slots in < 0.5ms total.
  - Boss detection must not iterate all entities — use spatial hashing or distance
    checks before performing expensive isBoss() evaluation.
  - Better Combat API calls should be cached per weapon type, not per stack per tick.
  - Log performance metrics in debug mode (classification time, scan time, etc.).

SAFETY:
  - Every cross-mod interaction MUST be wrapped in try-catch. Log the exception and
    fall back to the next detection layer. The mod must NEVER crash another mod.
  - Every reflection call MUST check for null returns and handle NoSuchMethodException,
    NoSuchFieldException, and IllegalAccessException.
  - Use @OnlyIn(Dist.CLIENT) annotations correctly — no server-side class references
    in client code.
  - Use @OnlyIn(Dist.DEDICATED_SERVER) for server-side boss AI code.

CODE QUALITY:
  - All new classes must have Javadoc explaining their purpose and usage.
  - Use final for all fields that don't change after construction.
  - Use records or immutable POJOs for data transfer objects.
  - Use streams and functional interfaces where they improve readability, but avoid
    streams in hot paths (stick to for-loops for per-tick operations).
  - Constant values should be in a Constants class or the config, NEVER magic numbers
    in logic code (except in tests).

COMPATIBILITY:
  - Target Minecraft Forge 1.20.1 (Forge version 47.x).
  - Java 17.
  - Must work with or WITHOUT any of the compatible mods installed.
  - Must work on both dedicated servers and integrated servers.
  - Client-side bot code must work with or WITHOUT OptiFine/FORGE (check for
    rendering differences that might affect input simulation).

FINAL NOTES:
  - This system is for a CONTENT mod, not a library. Keep the code practical and
    focused on gameplay behavior, not over-engineered abstractions.
  - Prioritize WORKING code over PERFECT code. A simple heuristic that works for
    90% of modded weapons is better than a complex system that breaks on edge cases.
  - When in doubt, log a debug message and fall back to safe default behavior.
  - Test with Cataclysm, Better Combat, Iron Spells, and at least 2 other weapon
    mods before considering the implementation complete.
  - The Madness effect should feel CHAOTIC and FUN for the player watching it.
    Bosses should be terrifying under Madness, not confusing or bugged.
